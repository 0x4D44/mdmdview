# Journal: Status Bar Narrow Width

**Date**: 2026-03-07
**Goal**: Stop status bar text from colliding in narrow windows and simplify the visible status copy.

## Log

### 2026-03-07 11:10
- Reproduced the issue from Arthur's screenshot by inspecting `render_status_bar` in `src/app.rs`.
- Confirmed the bar used one long left label plus one long right label with no adaptive truncation.
- Confirmed `Mode: Rendered` and `Elements: N` were purely status chrome and not essential for everyday use.

### 2026-03-07 11:25
- Switched the bar to a two-cell strip layout so left and right sections stop fighting over the same space.
- Added middle-ellipsis shortening for the file path, biased toward preserving the filename.
- Reduced visible stats to `Words`, `Flesch`, and `Chars`, with narrower fallbacks as space tightens.
- Kept view mode and parsed element count in the hover tooltip so the information still exists without polluting the bar.

### 2026-03-07 11:35
- Added unit coverage for the path ellipsis helper, status summary content, and a narrow-width status bar render.
- Feeling: this one was straightforward once the layout stopped pretending that free-form labels would self-regulate. The old bar was doing too much in one row.

### 2026-03-07 11:55
- Arthur requested the patch version bump too, so I moved `mdmdview` from `1.13.1` to `1.13.2`.
- Updated the root package version in `Cargo.toml` and the matching `Cargo.lock` package entry without touching unrelated lockfile content.

### 2026-03-07 12:15
- Follow-up issue: the new strip layout stopped pinning the filename to the left edge.
- Root cause was `StripBuilder` cell layout inheritance interacting with truncated labels.
- Fixed by forcing the status strip cells to use a left-to-right, non-justified layout.

### 2026-03-07 12:45
- That still did not fix the visible result in the live app.
- Reproduced with `mdscreensnap` against a real narrow window and then checked `egui` internals.
- Actual root cause: `ui.add_sized(...)` centers and justifies its child widget.
- Replaced the status-bar truncated labels with an explicitly left-aligned child UI instead of `add_sized`.

### 2026-03-07 13:20
- Added a Winget publish job to the release workflow.
- The new job runs after the GitHub release step and submits a manifest update using `wingetcreate`.
- It skips cleanly unless the repo has `WINGET_PACKAGE_IDENTIFIER` configured as a variable and `WINGET_CREATE_GITHUB_TOKEN` configured as a secret.
- Important caveat: this assumes the package already exists in Winget; first submission is still a manual bootstrap.

### 2026-03-07 13:35
- Arthur confirmed the Winget package already exists as `0x4D44.mdmdview`.
- Updated the release workflow so `workflow_dispatch` can target an existing tag instead of forcing a new version cut.
- Added a small resolve job that normalizes the effective tag/ref and makes the build, GitHub Release, and Winget steps all use the same target tag.

### 2026-03-07 13:55
- Checked GitHub: the original `v1.13.2` release workflow run succeeded and published the GitHub release assets.
- Found repo configuration mismatch: workflow expected `WINGET_CREATE_GITHUB_TOKEN`, but the repo currently has `WINGET_TOKEN`.
- Updated the workflow to accept either secret name and planned a manual rerun targeting `v1.13.2`.
