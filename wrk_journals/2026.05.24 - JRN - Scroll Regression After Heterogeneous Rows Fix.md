## Prior fix notes (from 2026.01.20 - JRN - scroll-bug.md)

Two phases of fix back then:

1. **Phase 1 (2026.01.20):** broaden initial row-height estimation for small tables, fix remainder column width estimation.
2. **Phase 2 (2026.01.21):** root cause was *cache propagation* — heights were stored in a local vector, but only rendered rows got `update_row_height` calls. On frame 2+, `needs_estimate` was false (because *some* rows had cache entries), so the remaining (unrendered) rows reverted to fallback heights. Fix: cache estimated heights immediately after calculation, not just after rendering.

That fix was reverted/subsumed when commit `62c2ad7` moved to uniform-row-height. My heterogeneous fix from yesterday brought back per-row heights and added the immediate-cache write back in. So phase-2 protection is in place. But that doesn't mean phase-2 was the *only* bug — phase-1 (estimation accuracy) still matters.

## Hypothesis 0 — from a quick read of yesterday's diff

The fix yesterday added:
```rust
self.update_row_height(table_id, idx, estimate);   // during estimation
...
self.update_row_height(table_id, idx, actual_row_height);   // during render
```

So a row that's visible on frame N has its cache OVERWRITTEN with `actual_row_height` (after measurement). On frame N+1, the cached value is the actual (not the estimate). For off-screen rows, cache still holds the estimate.

If the estimate over-estimates compared to actuals (likely, since estimate uses padding + line-height heuristics), then as rows scroll into view:
- before: cached estimate is X (let's say 60px)
- they render at actual Y (say 40px)
- cache updates to 40px
- total table height shrinks by 20px per row that scrolls into view

`heterogeneous_rows` calls our height iterator each frame. Total height = sum. If sum keeps shrinking as user scrolls, the outer `ScrollArea` sees content getting smaller, scroll position resets to top.

This matches the symptom: "scroll to 80%, jumps to top" — by the time you've scrolled past most rows, the total height has shrunk significantly from where it was at the start.

## Hypothesis 1 — confirmed analytically (without reproducing yet)

Tracing the heterogeneous-rows path in `render_table_tablebuilder`:

- Frame 1: `needs_estimate = true`. Estimate populates cache + the `row_heights` vector.
- Frame 1 render: `body.heterogeneous_rows(row_heights.into_iter(), ...)` allocates a slot per row using my iterator values. heterogeneous_rows accumulates `cursor_y` by `row_height + spacing.y` for **every** row in the iterator, regardless of visibility. So content_height = sum(row_heights).
- Inside the row closure for visible rows: render measures actual height, calls `update_row_height(idx, actual_row_height)`. This **overwrites** the cached estimate with the actual.
- Frame 2: `needs_estimate = false` (cache populated). We re-read cache to build the iterator. Visible-on-frame-1 rows have actuals; invisible-on-frame-1 rows still have estimates.
- If estimate over-predicts (estimate > actual), every newly-visible row shrinks the cache value → total content_height shrinks frame-to-frame as user scrolls.
- Eventually `scroll_offset > content_height` → outer `ScrollArea` clamps back to top.

This matches Arthur's symptom exactly: scroll to ~80%, jump back. Phase-2 protection (cache immediately during estimation) is in place, but it doesn't protect against the **over-estimate → shrink-on-measurement** drift.

### Why uniform-rows didn't have this bug

With the uniform approach (the code we just replaced), `update_row_height(idx, actual)` per-row still happened, but the next frame's iterator value came from:

```rust
(0..rows.len())
    .filter_map(|idx| ...row.max_height)
    .fold(fallback_row_height, f32::max)   // ← max across ALL rows
```

→ uniform height = max(cached heights). The tall outlier row dominates; small rows' cache shrinkage is hidden by the max. Stable, but it was *also* the cause of the giant-row bug we just fixed.

## Fix

**Monotonic-grow cache** in `update_row_height`:

```rust
fn update_row_height(&self, table_id: u64, idx: usize, height: f32) {
    let clamped = height.max(self.row_height_fallback());
    let mut metrics = self.table_metrics.borrow_mut();
    let entry = metrics.entry_mut(table_id);
    let row = entry.ensure_row(idx);
    row.max_height = row.max_height.max(clamped);   // ← was: = clamped
    row.dirty = false;
}
```

Semantics: estimates set the initial height; render measurements only update if they say the row needs **more** space, never less. Total content height never shrinks within a session → scroll position remains valid. Underestimate-then-grow case still triggers `height_change` → repaint → next frame's iterator picks up the larger value → content fits.

Trade-offs:
- Over-estimates leave a small empty band at the bottom of rows. Cosmetic only. The estimate uses the exact same `ui.fonts(|f| f.layout_job(...).size().y)` call as the renderer, so over-estimate should be small (mostly inter-fragment padding rounding).
- Cache is bounded to a single table-id session — when content changes, `table_id` changes (it's a content hash), so a new entry starts fresh. When font/zoom changes, `check_font_size_change` clears persisted widths and the existing width-change machinery should trigger re-estimation.
- This also stops the pathological `height_change → repaint → shrink → repaint` loop that could happen with the current code when actual < estimate.

## Cross-check vs Commit B (weighted Remainder)

The Commit B design needs the row-height estimator and the rendered widths to agree (Gap 1 I flagged earlier). Monotonic-grow doesn't change that requirement, but it does change my mental model:

- If Commit B writes weighted widths into `adjusted_widths`, both `estimate_table_row_widths` (height estimation) and `build_table_column_layout` (rendered widths) see the same values → estimates accurate → less reliance on monotonic-grow.
- Monotonic-grow is still the right fix for the *current* scroll bug because it removes the dependency on perfect estimate accuracy. Commit B's correctness becomes a "bonus" rather than a load-bearing requirement.

Both changes are complementary. Apply monotonic-grow first (fixes today's regression), then proceed with Commit B.

## Implementation

One-line change to `update_row_height` (`src/markdown_renderer.rs`):

```rust
row.max_height = row.max_height.max(clamped);   // was: row.max_height = clamped;
```

Plus a long comment block explaining the constraint. Plus a regression test (`test_update_row_height_monotonic_grow`) that verifies:
- Smaller measurement does NOT shrink the cache (the bug).
- Larger measurement DOES grow the cache (under-estimate recovery).
