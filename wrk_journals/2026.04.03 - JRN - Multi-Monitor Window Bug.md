# 2026.04.03 — Multi-Monitor Window Placement Bug

## Bug Report
User reports V1.13.4 handles docking to all edges of primary monitor correctly, but cannot
move the window to a second screen via Win+Arrow or dragging.

## Verdict: Valid Bug — Three Compounding Causes

### Cause 1: Runtime position clamping (PRIMARY)
**File:** `src/app.rs` — `compute_window_adjustment()` (line ~1176)

Runs every frame when not maximized/fullscreen. Computes bounds from `monitor_size` which is
a single monitor's dimensions (e.g. 1920×1080). Compares against `pos` which is in Windows
virtual desktop coordinates.

When dragging to a right-side monitor:
- `pos.x` becomes ~1920+ (virtual desktop coordinate)
- `max_pos.x` = `(1920 - window_width).max(0.0)` ≈ 1120
- Check: `pos.x (2000) > max_pos.x (1120) + 32` → true
- Result: `pos.x` clamped to 1120, window snaps back to primary monitor

The app literally fights the OS every frame. Same issue with Win+Arrow keys.

Also affects monitors to the left (negative X): check at line ~1240 clamps positions
more negative than -25% of window width back to 0.

### Cause 2: Startup restore clamps to positive
**File:** `src/window_state.rs` — `sanitize_window_state()` (line ~262)

```rust
pos: [ws.pos[0].clamp(0.0, MAX_POS), ws.pos[1].clamp(0.0, MAX_POS)],
```

Monitors left/above primary have negative virtual desktop coordinates (e.g. -1920, -1080).
This clamp forces them to 0.0, so even if you somehow got the window to a secondary monitor,
it would snap back to primary on restart.

### Cause 3: Registry stores positions as u32
**File:** `src/window_state.rs` — `save_window_state_registry()` / `load_window_state_registry()`

- Save: `v.max(0.0).min(u32::MAX as f32).round() as u32` — negative values become 0
- Load: `let x: u32 = key.get_value("PosX").ok()?` — can't represent negative

## Fix Plan

### Fix 1: compute_window_adjustment — stop fighting multi-monitor moves
Remove the position clamping entirely. Keep only:
- Non-finite coordinate recovery (NaN/Inf → 0,0)
- Size clamping (min/max dimensions still valuable)

Rationale: The OS handles window positioning correctly for multi-monitor. Our position
"safety" check can't distinguish "user moved window to second monitor" from "monitor was
disconnected" without knowing the full virtual desktop geometry, which egui doesn't expose.
The size checks remain useful (prevent windows smaller than 600×400 or larger than the
current monitor).

### Fix 2: sanitize_window_state — allow negative positions
Change clamp to `-MAX_POS..MAX_POS`. Negative positions are legitimate on multi-monitor.

### Fix 3: Registry — switch u32 to i32
Change PosX/PosY storage from `u32` to `i32` to handle negative virtual desktop coordinates.

### Tests
- Update existing sanitize tests for negative position support
- Add test: compute_window_adjustment with positions beyond single monitor bounds
- Add test: registry round-trip with negative coordinates (mock/unit level)

## Implementation Complete

### Changes Made

**`src/app.rs` — `compute_window_adjustment()`**
- Removed position boundary clamping (lines 1234-1255 in original)
- Removed unused `OFFSCREEN_TOLERANCE` constant
- Kept: NaN/Inf recovery, size min/max clamping
- Added comment explaining why position is intentionally unclamped

**`src/window_state.rs` — `sanitize_window_state()`**
- Changed `clamp(0.0, MAX_POS)` → `clamp(-MAX_POS, MAX_POS)` for both pos[0] and pos[1]

**`src/window_state.rs` — Registry read/write**
- Load: Read `u32`, reinterpret as `i32` via `as i32`, then `as f32`
- Save: Convert `f32` → `i32` (clamped), then `as u32` for DWORD storage
- Backward compatible: old positive values read identically through both paths
