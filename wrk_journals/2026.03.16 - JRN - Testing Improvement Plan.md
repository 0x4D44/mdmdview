# Journal: Testing Improvement Plan

**Date:** 2026-03-16
**Goal:** Comprehensive plan to address testing gaps identified in review

## 10:00 — Kickoff

Starting from the testing review that identified these opportunities:

1. Property-based testing with `proptest` for markdown parsing
2. Expand visual regression CI beyond single smoke test
3. Add contract-level integration tests
4. Test the test runner (`full_test.rs` at 48% coverage)
5. Snapshot testing with `insta` for parsed structures
6. Performance regression tracking
7. Mutation testing (`cargo-mutants`)
8. Fuzz testing (`cargo-fuzz`)

### Approach

Launching a 4-agent research team:
- **Agent A (Property/Fuzz):** Research proptest generators for markdown, cargo-fuzz setup, what invariants to check
- **Agent B (CI/Integration):** Research expanding visual regression CI, integration test design
- **Agent C (Infrastructure):** Research insta snapshots, cargo-mutants, performance regression
- **Agent D (Devil's Advocate):** Question assumptions, identify blind spots, challenge priorities

Will synthesize findings into a single plan document.

## 10:15 — Agent B (CI/Integration) reports back

**Critical finding:** There is no CI workflow that runs `cargo test` on PRs. The Rust test suite only runs in `release.yml`, which triggers on version tags. The `visual-regression.yml` runs on PRs but only does 1 of 26 visual cases — no Rust tests.

This means a PR can merge with broken tests and nobody knows until the next release tag. This should be **P0** — higher priority than any of the 8 original items.

Other key findings from Agent B:
- Expanding visual regression to all 26 cases is low-effort (remove `--case` filter), ~3-5 min extra CI time
- No cross-platform CI (claims Linux/macOS support, never tests it)
- `tests/api.rs` integration tests are feasible — most public API is headless-testable
- Mermaid visual in CI is hard (display dependency) but possible via `--screenshot` mode
- CI cost estimate: ~$0.30-0.40 per PR run on `windows-latest`

## 10:25 — Agent A (Property/Fuzz) reports back

Thorough analysis of the parsing pipeline. Key findings:

**Proptest:**
- `parse()` takes `&str`, returns `Result<Vec<MarkdownElement>>`. pulldown-cmark never errors; only forced test injection causes errors. So "parse never panics" is the #1 property.
- 10 specific properties identified, grouped by priority:
  - CRITICAL: parse never panics
  - HIGH: highlight_segments covers entire text, pipe escape preservation, text content preservation
  - MEDIUM: header ID uniqueness, element count bounded by input, idempotence
  - LOW: slugify/fold_for_search idempotence, line ending normalization
- `MarkdownRenderer::new()` is expensive (syntect, threads) — need `LazyLock` to share across proptest cases
- `parse()` does NOT need an egui context — proptests can run headless and fast

**Fuzz testing:**
- 5 primary targets: parse_markdown, escape_table_pipes, d2_parse, highlight_segments, slugify
- 4 secondary: normalize_line_endings, is_html_line_break, fold_for_search, pikchr_render
- **cargo-fuzz doesn't officially support Windows** — need WSL, or use `bolero` crate as engine abstraction
- D2 parser has no recursion depth limit — stack overflow risk on deep nesting
- highlight_segments does byte/char mapping with Unicode case folding (German ß→ss) — prime edge case territory

**Practical concern:** Adding proptest with 256 cases/property would add ~2-5 seconds total — negligible impact on test runtime.

## 10:35 — Agent D (Devil's Advocate) reports back

**This one changes the conversation.** Key challenges:

### Coverage numbers are misleading
- Main crate: 99.7% lines (real, but inflated by some truism tests)
- Workspace-wide including D2 crate: **90.3% lines, 86.6% branches**
- D2 crate: compiler 49.8%, shapes 54.9%, svg_render 74.7%
- Some tests explicitly named "cover noops" — testing that boolean assignment works

### The bugs we actually have aren't the bugs we're testing for
- Reviewed 92 bug-fix commits in project history
- Nearly all were: visual/layout issues, OS/platform interaction, filesystem edge cases, D2 geometry
- **None were caught by the test suite before production** — all were noticed in use
- Unit-level techniques (proptest, mutation testing, insta) don't address these categories

### Maintenance burden reality check
| Tool | 6-month prognosis |
|------|-------------------|
| proptest | Generators become stale |
| insta | `cargo insta accept` becomes muscle memory |
| cargo-mutants | Run once, never again |
| cargo-fuzz | Forgotten, run quarterly |
| Visual regression (26 cases) | Well-maintained (infrastructure exists) |

### "If you only do 3 things" shortlist
1. **Expand visual regression CI** from 1→26 cases (addresses actual bug category)
2. **Add cross-platform CI** (Linux job — not even on our list!)
3. **Write 20-30 targeted edge-case tests** (cheaper than proptest, more targeted)

### Items challenged as unnecessary
- Mutation testing: CPU-days for a solo developer, same insight as code review
- Testing the test runner: "turtles all the way down"
- Insta snapshots: fragile to any enum refactoring, high noise
- D2 crate should have separate visual-first testing strategy

This is a legitimate reframe. The original 8-item list was unit-testing-heavy for a project whose bugs are visual/environmental. Need to balance the final plan accordingly.

## 10:45 — Agent C (Infrastructure) reports back

Key findings:

**Snapshot testing (insta):**
- `MarkdownElement` and `InlineSpan` derive `Debug` but NOT `Serialize` — `assert_debug_snapshot!` works without adding serde
- ~30-40 of 476 tests have weak assertions (check variant exists, not content) — these are the snapshot candidates
- Full conversion would be noisy/fragile; selective use on weak tests is the sweet spot
- Verdict: Medium priority, use selectively

**Mutation testing:**
- Full codebase run would take CPU-days — impractical
- Scoped to D2 crate (~10K lines): feasible in 30-60 minutes
- Scoped to markdown_renderer.rs parsing: ~1-2 hours
- D2 crate is highest signal (most algorithmic, fewest tests per line)

**Performance regression:**
- **No benchmarks exist at all** — no benches/, no criterion, no #[bench]
- Parsing is O(n) via pulldown-cmark, not currently a problem
- Pathological inputs: thousands of nested blockquotes, hundreds of table columns, D2 Sugiyama with 200+ nodes
- Recommendation: criterion for 3-4 cases, track locally, don't use wall-clock assertions (flaky)

**D2 Sugiyama coverage:**
- **5 tests for 470 lines of graph algorithm** — dangerously thin
- `assign_coordinates` has zero direct tests
- `find_components` untested directly
- Needs 10-15 more targeted unit tests — this is a genuine gap

**Test runner (full_test.rs):**
- 48% coverage is appropriate — untested parts are orchestration that can't be unit-tested
- Testable pure functions (diff_images, html_escape) are already well-tested
- Verdict: Don't invest further

## 11:00 — Plan document written

Saved to `wrk_docs/2026.03.16 - Testing Improvement Plan.md`.

### Key decisions made during synthesis

1. **Reordered everything.** The original 8-item list was items 1-8. The final plan has 4 tiers with different priorities. Two items not on the original list (PR CI, cross-platform CI) became P0/P1.

2. **Scaled back mutation testing.** Agent C and D both flagged it as impractical at full scale. Kept only for D2 crate (small, algorithmic, diagnostic).

3. **Kept proptest but reduced scope.** Instead of 10 properties, focused on 3 must-haves: parse never panics, highlight_segments coverage, D2 parse never panics. These address genuine unknown-unknowns without massive maintenance burden.

4. **Honored the devil's advocate.** The challenge that "none of 92 bug-fix commits were caught by tests before production" is the most important insight. It means the #1 priority is testing the things that actually break (visual fidelity, platform behavior), not perfecting unit-level coverage.

5. **Treated D2 crate separately.** It's a graphics/layout library whose correctness is fundamentally visual. Coverage metrics are less meaningful; geometric invariants and visual comparison are the right oracles.

### What the plan does NOT do

- Chase coverage metrics for their own sake
- Add mutation testing or fuzzing to CI (too slow/expensive)
- Wholesale convert to insta snapshots (maintenance burden)
- Test the test runner further (48% is appropriate)
- Remove existing truism tests (not worth the churn)
