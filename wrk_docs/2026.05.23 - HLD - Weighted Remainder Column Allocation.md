## 1. Summary

When a markdown table has two or more `Remainder` columns, egui_extras splits leftover viewport space equally between them. In tables like `tested_dos_games.md`, this stretches narrow columns (e.g. a `Report` column containing only short links) to half the available width while genuinely content-heavy columns (e.g. `Notes`) wrap aggressively. This design replaces the equal-split with **proportional allocation by natural content width**: the column that needs more space gets more space.

## 2. Problem Statement

`build_table_column_layout` emits `Column::remainder()` for every `Remainder`-policy column. egui_extras' `Column::remainder()` divides leftover viewport space evenly across all such columns. There is no knob for weighting.

Visible failure mode on `D:\language\mddosem\tested_dos_games.md`:
- `Notes` column (some cells contain multi-paragraph error descriptions): ~290 px, wrapping to 10+ lines per cell.
- `Report` column (cells contain `[Report](...)` or `[See journal](...)`): ~510 px, mostly empty space.

A sibling fix (Commit A, already landed in `04fc97e`) stops misclassifying short-link columns as `Remainder`. This design handles the case where two columns *legitimately* want flex space but in unequal amounts.

## 3. Goals & Non-Goals

**Goals**
1. Multi-`Remainder` tables allocate space proportionally to natural content width.
2. Existing behavior preserved when there is one `Remainder` column, or when all `Remainder` columns have equal natural width.
3. Users gain drag-to-resize and persistence on previously-Remainder columns (side benefit of the implementation strategy).

**Non-Goals**
- No changes to `Fixed`, `Auto`, content-fit, or hscroll modes.
- No changes to column *classification* (which columns get which policy) — that is Commit A's territory.
- No new UI controls, no settings, no feature flags, no env vars.

## 4. Current State

`src/markdown_renderer.rs::build_table_column_layout`, viewport-fill branch, `ColumnPolicy::Remainder` arm:

```rust
ColumnPolicy::Remainder { clip } => {
    let mut col = if scaled_down {
        Column::initial(width).at_least(min_floor.min(width))
    } else {
        Column::remainder().at_least(min_floor)   // ← equal split
    };
    ...
}
```

`Column::remainder()` is opaque to caller-side weighting and is not user-resizable.

## 5. Proposed Design

### 5.1 Algorithm

In viewport-fill mode, when iterating columns:

1. Collect the `Remainder` column indices and compute `nat_total = Σ natural_widths[i]` over them.
2. Identify the **soak column**: the `Remainder` column with the largest natural width.
3. For each non-soak `Remainder` column:
   - `weight = natural_widths[i] / nat_total` (if `nat_total == 0`, fall back to `1 / count` — equal split).
   - `target = remaining_for_remainders * weight`.
   - `width = max(target, min_floor)`.
   - Emit `Column::initial(width).at_least(min_floor.min(width)).resizable(true)`.
4. For the soak column, emit `Column::remainder().at_least(min_floor)` — it absorbs rounding and any pixels left over.

Where `remaining_for_remainders = available_for_columns - fixed_total - non_remainder_flex_total` (currently computed implicitly inside `compute_table_layout_plan`; this design promotes it to a field on `TableLayoutPlan`).

### 5.2 Worked example

`tested_dos_games.md` at ~2000 px viewport, 7 columns. Suppose Fixed + non-Remainder flex consume 1200 px, leaving 800 px for two `Remainder` columns (Notes, Report). Natural widths from `estimate_natural_column_widths`: Notes ≈ 200 px, Report ≈ 80 px.

- `nat_total = 280`.
- Soak column = Notes (largest natural).
- Report: `weight = 80/280 ≈ 0.286`, `target = 800 * 0.286 ≈ 229 px`. Emit `Column::initial(229).resizable(true)`.
- Notes: emit `Column::remainder()` → gets the remaining 571 px.

Result ratio 571 : 229 ≈ 2.5 : 1, matching the 200 : 80 natural-width ratio. Compare to today's equal split of 400 : 400.

### 5.3 Edge cases

| Case | Behavior |
|---|---|
| 0 Remainder columns | Branch never entered; no change. |
| 1 Remainder column | Soak selection picks it; emits `Column::remainder()`. Identical to today. |
| All Remainder naturals equal | Weighting reduces to equal split. Backward-compatible. |
| `nat_total == 0` | Fall back to `weight = 1 / count` (equal). |
| `weighted < min_floor` | Clamp; soak column absorbs the deficit. |
| `scaled_down == true` | Use existing `scaled_down` path unchanged — weighting is for the expansion case only. |

### 5.4 Decisions

| Question | Decision |
|---|---|
| Soak column choice | Largest natural width. |
| Threshold for weighting | Applies for 2+ Remainder columns (no special-case for the 2-col case). |
| User-resize / persistence on previously-Remainder columns | Silently enabled. Users who don't drag see no change; users who do drag get persistence "for free" via the existing `Resizable` path. |

### 5.5 Why this is simple

- Two files touched (`src/markdown_renderer.rs`, plus tests).
- ~60 lines added across one helper and one match arm.
- No new traits, types, or modules.
- No new persistence path — the existing `Resizable` persistence handles the new draggable columns automatically.
- The fallback for the `nat_total == 0` and single-Remainder cases collapses to today's behavior, so partial application (during development) never breaks anything.

## 6. Implementation Sketch

Localized to `src/markdown_renderer.rs`:

1. **`TableLayoutPlan`** — add `remainder_indices: Vec<usize>` and `remaining_for_remainders: f32`. Both already computed inside `compute_table_layout_plan`; just promote them to fields.
2. **`compute_table_layout_plan`** — populate the new fields.
3. **`build_table_column_layout`** — in the viewport-fill `Remainder` branch, consult the new fields to pick the soak column and emit either `Column::initial(weighted).resizable(true)` or `Column::remainder()`.

Helper function (private, in same file):

```rust
fn weighted_remainder_widths(
    natural_widths: &[f32],
    remainder_indices: &[usize],
    remaining_for_remainders: f32,
    min_floor: f32,
) -> (usize /* soak_idx */, Vec<(usize, f32)> /* (col_idx, width) for non-soak */)
```

Pure function, trivially unit-testable.
