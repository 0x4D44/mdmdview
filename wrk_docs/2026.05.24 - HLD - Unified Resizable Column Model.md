# Unified Resizable Column Model

**Date:** 2026-05-24
**Author:** Arthur + Claude
**Status:** Draft

## 1. Problem

Table column width allocation uses a 4-policy model (Fixed, Resizable, Remainder, Auto) with a keyword-matching heuristic (`classify_column`) to assign policies from header names. This is brittle and overcomplex:

- A column named "Notes" gets `Resizable` via keyword match, preventing it from absorbing excess space even when its content is by far the longest.
- "Year" and "Report" columns get `Resizable` with generous preferred widths, then egui proportionally inflates them to fill the viewport.
- Fixed columns (`Column::exact`) cannot be resized by the user at all.
- The keyword cascade (version → date → author → status → description → example → first-column → content-driven → default) is fragile: it fires on substrings, the priority ordering creates surprises, and it guesses pixel widths from header names without looking at content.

The result on `tested_dos_games.md`: the Notes column (hundreds of characters per cell) gets ~180px while Year (4 digits) and Report ("See journal") each get ~400px.

## 2. Goal

Replace the 4-policy model with a single model: **all columns are resizable, sized by content, with excess viewport space distributed proportionally to content width**. Users can drag any column to resize. No clipping.

## 3. Design

### 3.1 The content-fit branch already works

`build_table_column_layout` has a `content_fits` branch (table content narrower than viewport) that ignores policy entirely:

```rust
Column::initial(natural_width).at_least(min_floor).resizable(true)
```

Every column gets content-driven width, every column is draggable. This is the model we want everywhere.

### 3.2 Extend to viewport-fill

When table content exceeds the viewport, distribute available space proportionally to natural content width:

1. Compute `natural_widths[i]` per column (unchanged — max_graphemes × char_width + padding).
2. Compute `nat_total = Σ natural_widths[i]`.
3. For each column: `adjusted_width[i] = available_for_columns × (natural_widths[i] / nat_total)`.
4. Clamp each to `at_least(min_floor)`. Last column absorbs rounding residual.
5. Emit `Column::initial(adjusted_width[i]).at_least(min_floor).resizable(true)`.

This is the weighted Remainder allocation from the previous HLD, applied to *all* columns.

### 3.3 Scale-down

When total minimum widths exceed available space: proportional scale-down toward minimums (existing logic), applied uniformly. No policy-specific branching.

### 3.4 Minimum widths

All columns share a content-derived minimum:

- `min = max(min_floor, header_width_px)` — never narrower than the header text or the global floor.

This replaces the per-keyword min calculations with a single rule.

### 3.5 No clipping

Drop `clip` entirely. All columns wrap instead of truncating. Simpler model — one fewer concept. Users can drag columns wider if wrapping is unwanted.

### 3.6 User resize persistence

All columns are now resizable, so all get persistence. The `persist_resizable_widths` guard on `ColumnPolicy::Resizable` is removed.

Persisted widths override proportional allocation: if the user has manually dragged a column, that width is used instead of the proportionally-computed one. On first load (no persisted widths), proportional allocation drives everything.

### 3.7 `resolved_widths` simplification

Currently `estimate_natural_column_widths` computes `max(content_width, policy_width)` where `policy_width` comes from keyword-derived preferred widths. Under the unified model there are no policy-derived widths. The pipeline simplifies:

```
natural_widths (content-driven)
    → apply persisted user overrides (if any)
    → proportional allocation to fill viewport
    → adjusted_widths
```

`resolve_table_column_widths` either returns natural widths (first load) or persisted widths (after user drag). One step fewer than today.

### 3.8 Natural width cap

`estimate_natural_column_widths` clamps at `body * 35.0` (~490px). This cap compresses the proportion ratios — a 200-char column and a 500-char column both cap at 490px, getting equal proportions.

This is acceptable and arguably beneficial: it prevents one column from hogging the entire viewport. At any given width, you can only see a limited amount of text regardless, so equal-ish allocation above the cap is reasonable. The cap already exists and we leave it unchanged.

### 3.9 Worked example

`tested_dos_games.md` PARTIAL table at ~1360px viewport, 6 columns, spacing ~6px per gap (5 gaps = 30px). Available for columns: ~1330px.

| Column | max_graphemes | natural_width (est) | Proportion | Allocated |
|--------|---------------|---------------------|------------|-----------|
| Game | 18 | ~181px | 16% | ~213px |
| Year | 4 | ~60px | 5% | ~67px |
| Publisher | 20 | ~199px | 18% | ~239px |
| Notes | 200+ | ~490px (capped) | 44% | ~585px |
| Test Date | 10 | ~112px | 10% | ~133px |
| Report | 15 | ~155px | 14% | ~186px |

*Proportions recomputed without the `max(content, policy)` uplift from resolved_widths.*

Compare to current: Notes ~180px, Year ~110px, Report ~400px. Notes goes from least space to most space. Year shrinks from ~110px to ~67px (still plenty for 4 digits).

## 4. What changes

### 4.1 `column_spec.rs`

- **`ColumnPolicy` enum**: remove `Fixed`, `Remainder`, `Auto` variants. Either collapse to a single-variant enum or remove entirely — the only remaining data is `min: f32` on `ColumnSpec`.
- **`classify_column`**: delete entirely. No keyword matching.
- **`column_needs_remainder`**: delete.
- **`derive_column_specs`**: simplify to a loop that computes `min` per column from header text width. No remainder tracking, no fallback promotion, no scored indices.
- **`MAX_REMAINDER_COLUMNS`**: delete.
- **`column_tooltip`**: simplify (one policy type, or remove tooltips).
- **`ColumnPolicy::to_column()`**: simplify to always emit `Column::initial(preferred).at_least(min).resizable(true)`.
- **`calculate_policy_hash`**: simplify (no discriminant branching needed).

### 4.2 `markdown_renderer.rs`

- **`compute_table_layout_plan`**: the viewport-fill branch simplifies. No fixed_total / flexible_indices / remainder_indices separation. All columns participate in a single proportional allocation from natural_widths.
- **`build_table_column_layout`**: viewport-fill collapses to match content-fit — `Column::initial(adjusted).at_least(min).resizable(true)` for every column. No clip.
- **`resolve_table_column_widths`**: simplify. No per-policy branching. Returns persisted width if available, otherwise natural width.
- **`persist_resizable_widths`**: remove the `ColumnPolicy::Resizable` guard — persist all columns.
- **`estimate_natural_column_widths`**: remove the `max(content_width, policy_width)` — natural width is purely content-driven. The `resolved_widths` parameter may go away.
- **`estimate_table_row_widths`**: unchanged (already returns adjusted_widths).
- **`apply_persisted_widths`**: `apply_preferred_width` currently only updates `Resizable`. Under unified model, updates all columns.

### 4.3 Tests (~30 affected)

- **~28 classification tests** in `column_spec.rs`: rewrite for the new model (fewer tests, simpler assertions — just check min and that all columns are resizable).
- **6 tests** using `ForcedTablePolicies` in `markdown_renderer.rs`: update or simplify. The infrastructure itself may reduce to forcing a single policy type.
- **Render tests** (cell content, no panics, scroll behavior): should pass unchanged — they test behavior, not policy assignment.

## 5. What doesn't change

- `estimate_natural_column_widths` — content-driven width calculation stays (minus the policy_width uplift).
- `compute_column_stats` / `accumulate_stats_for_cell` — content scanning stays.
- `ColumnStat` / `RichContentFlags` — content stats stay (needed for natural width).
- Content-fit mode — already correct, becomes the template for viewport-fill.
- Horizontal scroll mode — triggered by the same condition.
- Row height estimation — reads from adjusted_widths (unchanged).
- Monotonic-grow cache — still needed for scroll stability.
- Table metrics / width persistence infrastructure — stays.

## 6. Risks

| Risk | Mitigation |
|------|------------|
| Short tables (2 cols, few chars) look odd with proportional fill | Content-fit mode handles these — table won't fill viewport if content is short |
| Losing keyword-derived mins (e.g., date columns had px(8.5) min) | Header-width-based min is a better floor — "Test Date" header text is wider than "1994" content |
| Proportional allocation gives tiny columns when one column dominates | min_floor clamp prevents columns from becoming unusably narrow |
| Cap (~490px) compresses proportions for very long columns | Acceptable — prevents one column from hogging viewport; two capped columns get equal share |
| Test churn (~30 tests) | Policy tests are in column_spec.rs; render tests are behavior-based and survive |
| Image columns may get underweighted | Pre-existing — images scale at render time; natural_widths use text metrics. Not worse than today |
