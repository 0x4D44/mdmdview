## Goal

Push the main application source set toward MC/DC-style decision coverage without degrading the codebase.

Hard target:

- preserve `>= 98%` main-app branch coverage

Practical stretch target:

- reach `99.0%` to `99.5%` main-app branch coverage

Non-goal:

- literal 100% at any cost

Reason:

- with AI-assisted test generation, many decision-matrix tests are now cheap enough to justify
- the expensive part is no longer typing tests; it is preserving good oracles and avoiding brittle seams

## Planning Principles

1. Prefer MC/DC-style matrices for decision-heavy helpers and state machines.
2. Favor pure helpers and explicit state transitions over GUI event plumbing.
3. Do not contort production code just to fake OS behavior unless the path has real bug value.
4. Treat defensive, effectively unreachable branches as candidates for explicit triage, not automatic conquest.
5. Re-measure after each stage and stop when the remaining misses are mostly toolkit or platform noise.

## Feasibility Estimate

Rough estimate for the remaining `49` uncovered branch outcomes:

- `35-40` are feasible and worth doing
- `6-10` are technically feasible but likely brittle or low-signal
- `3-5` are probably not worth chasing without artificial seams or selective coverage exclusions

This is why the plan targets `99.0%` to `99.5%` first. That range captures most of the meaningful decision logic without turning the code into coverage theater.

## Stage 2 - Markdown Renderer MC/DC: Image Cache and Staleness

Objective:

- cover the high-value cache and stale-image decision logic in `markdown_renderer.rs`

Primary decision areas:

- `ImageCache::insert` and empty-order capacity behavior
- `ImageCache::enforce_budget` phase transitions and early-break conditions
- image loader failure-path classification
- degraded-restore accounting
- stale timestamp handling, including clock-backwards logic

Expected yield:

- about `10-12` branch outcomes
- likely branch coverage result: roughly `98.6%` to `98.8%`

Concrete work:

- write table-driven tests for `enforce_budget` varying:
  - total bytes
  - max bytes
  - number of non-degraded entries
  - near/far viewport distances
  - pending restore bytes
- add direct tests for image loader failure branches that still collapse to `Failed { key, generation }`
- add a deterministic stale-clock test for the `duration_since` error branch
- verify degraded-restore bookkeeping with explicit before/after byte assertions

MC/DC angle:

- each boolean condition should independently flip the outcome while other conditions remain fixed
- avoid giant integration fixtures when a direct helper-level test can prove the same decision

Exit criteria:

- most remaining cache/staleness branches are gone
- the remaining markdown gaps are mainly emoji/table/UI branches
