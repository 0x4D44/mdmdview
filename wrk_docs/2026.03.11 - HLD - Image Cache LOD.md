# HLD - Image Cache LOD (Level-of-Detail)

## Problem

When a document's decoded images exceed the 100 MB cache budget, the current
viewport-aware eviction strategy removes far-from-viewport entries to make room.
This has two unintended side effects:

1. **Layout shift**: Removing a cached image replaces it with a text placeholder
   ("Loading image...") that is a different height. This changes the document
   length and moves the scroll position, potentially shifting the viewport.

2. **Flicker feedback loop**: The viewport shift can bring evicted images back
   into view (or push near-viewport images out), triggering a new round of
   eviction, producing a sustained flicker cycle.

Both problems stem from the same root cause: eviction changes the rendered size
of image elements, which destabilises layout.

## Goal

Maintain a stable document layout at all times, even when images exceed the
cache memory budget. Images far from the viewport should consume negligible GPU
memory. Images near the viewport should be displayed at full resolution.

## Design

### Core idea

Never remove an image from the cache. Instead, **degrade** far-from-viewport
images to a tiny (2x2 pixel) solid-colour placeholder texture while preserving
the display dimensions (`size` field) in the cache entry. The rendering path
continues to size the `egui::Image` widget using these dimensions, so layout is
unchanged. When a degraded image scrolls back near the viewport, enqueue a
background reload to restore the full-resolution texture.

### ImageCacheEntry changes

```rust
struct ImageCacheEntry {
    texture: egui::TextureHandle,
    size: [u32; 2],           // display dimensions (post-downscale; never changes during degradation)
    modified: Option<SystemTime>,
    byte_size: usize,         // current texture memory (tiny when degraded)
    degraded: bool,           // true when texture is a 2x2 placeholder
}
```

**Important**: `size` stores the *display* dimensions — i.e. the dimensions
after `downscale_color_image` has been applied.  These are the dimensions the
rendering code uses for `fit_to_exact_size`.  They are **not** the original
decoded dimensions (which may be larger).  During degradation, `size` is left
unchanged so layout is preserved.

`byte_size` tracks actual GPU memory. When an entry is degraded, `byte_size`
drops to `2 * 2 * 4 = 16` bytes. When restored, it returns to the full value.
`total_bytes` stays accurate because `degrade()` adjusts it.

### ImageCache changes

Remove eviction from `insert()`. Replace it with a separate `enforce_budget()`
method that degrades entries instead of removing them:

```
fn enforce_budget(&mut self, ctx: &egui::Context, viewport_distances: &HashMap<String, f32>)
```

This method runs in two phases:

**Phase 1 — Orphan entries**: Degrade non-degraded entries that have NO viewport
distances. These are leftovers from a previous document and are safe to degrade
before touching current-document entries. This prevents stale entries from
hogging the cache budget.

**Phase 2 — Current-document entries**: Collect non-degraded entries with
viewport distances. Sort by distance descending (furthest first). Walk the
list, degrading entries until `total_bytes <= max_bytes` or only the nearest
entry remains.

Degradation replaces the entry's `TextureHandle` with a 2x2 neutral grey
(`#808080`) `ColorImage` allocated via `ctx.load_texture()`, updates
`byte_size` to 16, subtracts the difference from `total_bytes`, and sets
`degraded = true`.

Why a neutral grey:
- Visible if accidentally displayed at full size (debugging aid).
- Doesn't flash white or black during the reload frame.
- Good enough for v1; per-image average colour is a future refinement.

### Rendering changes

The image rendering path (`InlineSpan::Image` in `render_inline_span`) already
uses `entry.size` for layout via `egui::Image::fit_to_exact_size`. No layout
change is needed.

The reload decision lives in `get_or_load_image_texture`. When it finds a
cached entry with `degraded == true`:

1. It still returns `Some((texture, size[0], size[1]))` — the 2x2 placeholder
   texture with the original display dimensions. Layout is preserved.
2. It checks the *previous frame's* viewport distance for this key (stored in
   `ImageCache::viewport_distances`). One frame of lag is invisible.
3. If the entry is within the reload zone, not already pending, and the budget
   check passes (see Reload-zone logic below), it enqueues a background reload.

The degraded entry continues to be rendered while the reload is in flight.
This means the user sees a grey rectangle briefly, not a layout-shifting
placeholder. When the reload completes, `insert()` replaces the entry with
the full-resolution texture (`degraded` defaults to `false` on new entries).

### Calling enforce_budget

`enforce_budget()` runs once per frame at the end of `render_to_ui()`, after
all image Y positions have been collected and viewport distances computed. This
is the same place where `update_viewport_distances()` is currently called.

Sequence per frame:

1. `poll_image_results()` — process completed background loads (may restore
   degraded entries).
2. Render all elements. For each image, `get_or_load_image_texture` returns
   whatever texture is cached (full-res or degraded 2x2). If degraded and
   within reload zone and budget allows, a background reload is enqueued here.
3. Collect `image_doc_y` positions.
4. Compute viewport distances.
5. Call `enforce_budget()` — may degrade far-from-viewport entries.

The degraded textures take effect on the *next* frame. This is fine because
egui is immediate-mode; one frame of stale texture is invisible to the user.

### insert() simplification

`insert()` no longer needs eviction logic. It simply:

1. If the key exists, update the entry (adjust `total_bytes`).
2. Otherwise, add the new entry.

Budget enforcement is handled separately by `enforce_budget()`. This makes
`insert()` straightforward and removes `pick_eviction_candidate()`.

### Reload-zone logic

When rendering a degraded image, decide whether to reload:

```
let in_reload_zone = viewport_distance < viewport_height * 2.0;
```

Where `viewport_distance` is the absolute distance from the image's Y centre to
the viewport centre, and `viewport_height` is the visible scroll area height.

Only enqueue a reload if:
- The entry is degraded.
- It is within the reload zone.
- It is not already pending (`image_pending`).
- **Restoring it would fit in the budget**: `total_bytes - entry.byte_size +
  estimate_bytes(entry.size) <= max_bytes`. Without this check, restored
  entries would be immediately re-degraded by `enforce_budget`, creating a
  degrade → restore → degrade thrashing loop.

This prevents reload storms: images far from the viewport stay degraded, and
images that can't fit in the budget also stay degraded.

### Blurry-edges policy

If all images in the document are within the reload zone and their combined
decoded size exceeds the cache budget, some must stay degraded. The rule:

- `enforce_budget()` sorts by viewport distance and degrades furthest first.
- The reload-zone budget check (above) prevents reloading entries that would
  immediately be re-degraded. This is the primary anti-thrash mechanism — no
  separate per-frame hysteresis tracking is needed.
- The result is that the furthest visible images stay as grey placeholders,
  which is stable (no cycling).

This is the "blurry edges" policy: the user sees full-resolution images near
the centre of their viewport, and grey placeholders at the edges. As they
scroll, the zone shifts — images entering the centre restore, images leaving
degrade. Layout never changes.

### Staleness and F5 reload

- **File modification checks on degraded entries**: Currently,
  `get_or_load_image_texture` removes stale entries from the cache
  (`cache.remove()`). For a degraded entry whose source file changes on disk,
  removing it would lose layout stability. Instead: if the entry is degraded
  and stale, mark it for reload (same as the normal degraded-reload path) but
  do **not** remove it. The degraded texture stays as the layout placeholder
  until the reload completes.
- **File modification checks on full-res entries**: Keep the existing behaviour
  (remove and re-enqueue). A momentary text placeholder from a genuine file
  change is acceptable — it's a one-shot event, not a feedback loop.
- **F5 reload**: `reset_image_state()` bumps the generation counter. All
  pending reloads are discarded. Degraded entries survive — they have valid
  (tiny) textures that maintain layout. The next frame's `enforce_budget()` pass
  sees them as degraded and, for those in the reload zone, enqueues fresh loads.

### What gets removed

- `pick_eviction_candidate()` — no longer needed.
- Eviction loop in `insert()` — replaced by `enforce_budget()`.

### What stays

- `order: VecDeque<String>` — still needed for the entry-count safety valve
  (see below). Keeps insertion order. `touch()` is no longer needed and can be
  removed (LRU promotion is irrelevant when we degrade instead of evict).

### Entry count limit

The current 256-entry capacity limit stays as a safety net. If entry *count*
(not bytes) exceeds capacity, `insert()` removes the oldest entry by insertion
order (`order.front()`). This handles pathological cases like a document
referencing thousands of tiny images. In normal use, the byte budget is the
binding constraint, not the entry count.

## Risks

- **One frame of grey**: When an image is first degraded, the next frame shows
  a grey rectangle before the full texture is restored. This is only visible for
  images near the viewport edge, and only for one frame. Acceptable.
- **Reload bandwidth**: If the user scrolls quickly through a long document,
  many images may need reloading. The existing background loader and generation
  counter handle this — stale results are discarded, and the pending set
  prevents duplicate requests.
- **egui texture memory**: `ctx.load_texture()` for the 2x2 placeholder
  allocates a small CPU-side copy inside egui. With 256 max entries, this is
  at most 256 * 16 bytes = 4 KB. Negligible.
- **`release_gpu_textures()`**: This existing method clears the entire cache
  (called on context recreation). After LOD, it still clears everything —
  degraded entries included. The next render pass shows text placeholders
  briefly and re-loads from disk. This is the same as today and acceptable
  because context recreation is rare.

## Non-goals

- Per-image average colour for placeholders (future refinement).
- Progressive/multi-level LOD (e.g. thumbnail → medium → full). The 2x2 → full
  jump is sufficient because the degraded state is only visible briefly.
- Viewport-predictive prefetching (loading images ahead of scroll direction).
