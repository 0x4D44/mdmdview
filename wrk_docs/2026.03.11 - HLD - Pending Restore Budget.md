# HLD - Pending Restore Budget

## Problem

When multiple degraded images are in the reload zone, each independently
passes the `would_restore_fit` budget check — because the check uses the
current `total_bytes`, which doesn't account for other reloads already
in flight.

Sequence:
1. Three images degraded. `total_bytes` = 85 MB.
2. `get_or_load_image_texture` runs for each. Each calls `would_restore_fit`:
   - A: 85 - 0 + 5 MB = 90 ≤ 100. Pass.
   - B: 85 - 0 + 5 MB = 90 ≤ 100. Pass.
   - C: 85 - 0 + 5 MB = 90 ≤ 100. Pass.
3. All three reloads enqueued. All complete. `total_bytes` → 100 MB.
4. `enforce_budget` degrades one or more. `total_bytes` drops.
5. Next frame: repeat.

This produces a degrade→restore→degrade oscillation visible as cache usage
jumping (e.g. 95↔100 MB) and scrollbar jitter on every repaint. Mouse
movement triggers repaints (egui-winit hardcodes `repaint: true` for every
`CursorMoved` event), making the oscillation continuous while the mouse moves.

## Fix

Add a `pending_restore_bytes` counter to `ImageCache`. It tracks the
estimated bytes of all in-flight restore reloads — enqueued but not yet
completed. `would_restore_fit` includes this counter so that the second
and third enqueues see the budget already claimed by earlier ones.

### ImageCache changes

One new field:

```rust
pending_restore_bytes: usize,
```

Initialised to 0. Cleared by `clear()`.

### Increment: when a restore reload is enqueued

In `get_or_load_image_texture`, after `enqueue_degraded_reload` succeeds
(i.e. `enqueue_image_job` returns `Ok`), add the estimated restore bytes
to the cache:

```rust
let restore_bytes = ImageCacheEntry::estimate_bytes(entry.size);
cache.pending_restore_bytes += restore_bytes;
```

This requires knowing `entry.size` at the enqueue site. We already have it
(from the cache lookup earlier in the function). Thread it through to the
enqueue call, or compute it before dropping the cache borrow.

If `enqueue_image_job` fails (channel full), don't increment — no reload
was started.

### Decrement: when a current-generation result arrives

In `poll_image_results`, when processing a **current-generation** result
for a key that was a degraded restore (entry exists in cache and is marked
`degraded`), subtract the estimated bytes. This applies to both `Loaded`
and `Failed` outcomes.

```rust
// After the generation check has passed, before inserting:
if cache.entries.get(&key).is_some_and(|e| e.degraded) {
    let est = ImageCacheEntry::estimate_bytes(cache.entries[&key].size);
    cache.pending_restore_bytes = cache.pending_restore_bytes.saturating_sub(est);
}
```

**Do NOT decrement for stale-generation discards.** The counter was already
zeroed by `reset_image_state()` when the document switched. Decrementing
stale results would incorrectly reduce pending bytes that belong to the
new document's reloads.

Check whether the entry is degraded before decrementing. First-time loads
(entry not in cache or not degraded) don't have pending restore bytes.

### Zero: on document switch

`reset_image_state()` zeroes `pending_restore_bytes` on the cache. All
pending reloads become stale (generation mismatch) and will be discarded,
so the counter should reflect zero in-flight restores.

### Updated would_restore_fit

```rust
fn would_restore_fit(&self, key: &str) -> bool {
    let entry = match self.entries.get(key) {
        Some(e) if e.degraded => e,
        _ => return false,
    };
    let full_bytes = ImageCacheEntry::estimate_bytes(entry.size);
    self.total_bytes.saturating_sub(entry.byte_size)
        + full_bytes
        + self.pending_restore_bytes
        <= self.max_bytes
}
```

With the counter, the three-image example becomes:
- A: 85 + 5 + 0 pending = 90 ≤ 100. Pass. Enqueue. pending = 5 MB.
- B: 85 + 5 + 5 pending = 95 ≤ 100. Pass. Enqueue. pending = 10 MB.
- C: 85 + 5 + 10 pending = 100 ≤ 100. Pass. Enqueue. pending = 15 MB.
- D (if there were one): 85 + 5 + 15 = 105 > 100. Fail. Not enqueued.

If total images exceed budget, the last ones are blocked. On the next
frame, restored images increase `total_bytes` and `enforce_budget`
degrades the furthest. The counter prevents re-enqueuing them immediately
because `pending_restore_bytes` is only decremented after the result
arrives. Stable.

## What doesn't change

- `enforce_budget` — unchanged. It degrades based on actual `total_bytes`.
- `degrade()` — unchanged.
- `insert()` — unchanged. It doesn't touch `pending_restore_bytes`.
- The frame sequence — unchanged.
- Status bar / tooltip — shows `total_bytes` which is actual, not projected.

## Edge cases

- **Estimate vs actual mismatch**: The estimated restore bytes
  (`estimate_bytes(entry.size)`) may differ slightly from the actual
  inserted bytes if the screen size changed between enqueue and
  completion (different downscale target). The error is bounded by one
  image's worth of bytes and self-corrects when the counter is
  decremented. Acceptable.

- **Channel full**: `enqueue_image_job` fails → don't increment. The
  entry stays degraded and retries next frame.

- **Stale result discard**: Generation mismatch in `poll_image_results` →
  do NOT decrement. `reset_image_state()` already zeroed the counter.
  Decrementing here would steal budget from the new document's reloads.

- **Failed load**: Image file deleted while degraded → `Failed` result →
  decrement. Entry stays degraded, `note_image_failure` applies backoff.
