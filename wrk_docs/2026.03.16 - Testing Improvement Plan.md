# Testing Improvement Plan

**Date:** 2026-03-16
**Project:** mdmdview
**Status:** Draft

## Executive Summary

A four-agent research team investigated eight proposed testing improvements. The devil's advocate agent reframed the discussion: **the project's actual bugs are visual, environmental, and platform-specific — not logic bugs**. Of 92 bug-fix commits in project history, none were caught by the existing test suite before reaching production. The original 8-item list was unit-testing-heavy for a project whose real risks lie elsewhere.

This plan reorders priorities based on where bugs actually come from, not where coverage metrics point.

## Tier 1 — Address Real Gaps

These items address genuine gaps that would catch real bugs. Each one corresponds to a category of bug the project has actually experienced.

### 1.1 PR CI Workflow

**Priority:** P0 — Critical
**Effort:** Low (1-2 hours, copy from release.yml)
**Impact:** Tests currently don't run on PRs at all

Create `.github/workflows/pr.yml`:
- Trigger: `pull_request` to `master`
- Steps: `cargo fmt --check`, `cargo clippy --all-targets -- -D warnings`, `cargo test --all-targets`
- Runner: `windows-latest`
- Estimated CI time: ~10-12 minutes

This is the single most impactful change. Every other improvement is moot if tests don't run before code merges.

### 1.2 Expand Visual Regression CI

**Priority:** P1 — High
**Effort:** Low (remove `--case 001-headings` filter, adjust thresholds)
**Impact:** Catches rendering regressions across 26 test cases instead of 1

The infrastructure already exists. Changes needed:
- Remove `--case 001-headings` from the smoke step in `visual-regression.yml`
- Bundle test fonts via `--test-fonts` to mitigate CI font differences
- Widen diff thresholds slightly for CI (e.g., `max_pixels: 1000`, `max_percent: 0.15`)
- Add CI-specific threshold overrides (environment variable or manifest flag)

Estimated additional CI time: +3-5 minutes (total ~8-10 minutes)

Most project bugs are visual. This directly addresses the #1 bug category.

### 1.3 Cross-Platform CI

**Priority:** P1 — High
**Effort:** Low (add job to PR workflow)
**Impact:** Catches platform-specific issues (WSL path bugs, DLL dependencies, filesystem timing)

Add an `ubuntu-latest` job to the PR workflow:
- `cargo fmt --check`
- `cargo clippy --all-targets -- -D warnings`
- `cargo test --all-targets`

This runs in parallel with the Windows job, adding no wall-clock time. Linux runners are 1x billing vs Windows 2x.

Real bugs this would have caught: WSL image flicker, VC runtime DLL dependency issues, filesystem timing assumptions.

### 1.4 D2 Sugiyama Algorithm Tests

**Priority:** P1 — High
**Effort:** Low (half day, no new tools)
**Impact:** 5 tests for 470 lines of graph layout algorithm is dangerously thin

Current coverage gaps:
- `assign_coordinates()` — zero direct tests
- `find_components()` — untested directly
- `layout_component()` — no isolated test
- Horizontal directions (`Right`, `Left`, `Up`) — untested in Sugiyama module

Add 10-15 targeted unit tests:
- Diamond-shaped DAGs (rank assignment with convergence)
- Large fan-out (one node → many children)
- Disconnected components with varying sizes
- Back-edge reversal ordering
- Barycenter tie-breaking
- Coordinate centering with containers
- Each layout direction

## Tier 2 — High-Value Targeted Additions

These provide meaningful quality improvement with reasonable maintenance cost.

### 2.1 Targeted Edge-Case Tests (Markdown Parsing)

**Priority:** P2
**Effort:** Medium (1-2 days)
**Impact:** More targeted and cheaper than proptest for known weak spots

Write 20-30 hand-crafted tests for markdown-to-element conversion edge cases:
- Deeply nested lists (10+ levels)
- Tables inside blockquotes
- Links containing special characters (`()`, `[]`, backticks)
- Mixed line endings inside code blocks
- Extremely long single lines (10K+ characters)
- Tables with 50+ columns
- Empty tables (headers only, no rows)
- Inline code containing pipe characters in tables
- Nested emphasis in links (`[**bold *and italic***](url)`)
- Markdown with BOM, null bytes, lone surrogates (lossy UTF-8)
- Headings at every level with identical text (ID dedup stress)
- Code blocks with language strings containing spaces/special chars
- Adjacent horizontal rules
- Blockquotes containing tables containing code blocks

These exploit knowledge of the parser's weak spots from actual usage. They're cheaper to write, cheaper to maintain, and more targeted than property-based generation.

### 2.2 Property-Based Testing (Selective)

**Priority:** P2
**Effort:** Medium (1-2 days)
**Impact:** Finds unknown unknowns in parsing; negligible runtime overhead

Add `proptest` as a dev-dependency. Implement 3-5 high-value properties (not all 10 identified):

**Must have:**
1. **Parse never panics** — For any `String`, `renderer.parse(&s)` returns `Ok(_)`. This is the single highest-value property. pulldown-cmark won't panic, but the post-processing pipeline (`escape_table_pipes_in_inline_code`, `parse_inline_spans_with_breaks`) does manual index arithmetic.
2. **Highlight segments cover entire text** — `highlight_segments(text, needle)` returns non-overlapping segments spanning `[0, text.len())`. Unicode case folding (German `ß` → `ss`) makes this a prime edge case target.
3. **D2 parse never panics** — Pure function, trivial to test, recursive descent parsers benefit enormously. Also tests for stack overflow on deeply nested input (no recursion depth limit exists).

**Nice to have:**
4. Text content preservation (plain text extracted from parse output contains all input words)
5. `fold_for_search` and `slugify` are idempotent

**Practical notes:**
- `MarkdownRenderer::new()` is expensive (loads syntect, spawns threads). Use `std::sync::LazyLock` to share one instance across all proptest cases.
- `parse()` does NOT need an egui context — proptests run headless.
- Default 256 cases per property adds ~2-5 seconds total. Negligible impact on 8-minute suite.

### 2.3 Integration Tests

**Priority:** P2
**Effort:** Medium (1 day)
**Impact:** Validates public API contract; catches accidental export breakage

Create `tests/api.rs` exercising the public API as an external consumer:

1. **Markdown parsing round-trip** — Parse various inputs via `MarkdownRenderer::parse()`, verify element structure, verify `elements_to_plain_text()` output
2. **Sample file integrity** — Parse each `SAMPLE_FILES` entry, verify no errors
3. **Window state serialization** — `WindowState` save/load round-trip with tempdir
4. **App construction** — `MarkdownViewerApp::new()` succeeds, `load_content()` works, `close_current_file()` clears state
5. **Theme consistency** — `ThemeColors::current(true)` differs from `current(false)`

Most public API is headless-testable. Anything needing `egui::Context` or `egui::Ui` is out of scope for integration tests.

## Tier 3 — Useful, Lower Priority

These provide value but with diminishing returns or higher maintenance cost.

### 3.1 Selective Snapshot Testing (insta)

**Priority:** P3
**Effort:** Low-Medium (half day)
**Impact:** Strengthens ~30-40 tests that currently check variant existence but not content

Add `insta` as a dev-dependency. Use `assert_debug_snapshot!` (requires only `Debug`, already derived on all types — no `Serialize` needed).

**Where to apply:**
- Tests that use `matches!(el, MarkdownElement::List { .. })` without checking list content
- Tests that set `saw_code = true` but never check the code text
- Tests that check `elements.len()` without verifying content

**Where NOT to apply:**
- Anything involving `HighlightedToken` (contains `egui::Color32`, too noisy)
- Syntax highlighting output (changes with syntect theme updates)
- Tests that already make precise `assert_eq!` assertions

**Maintenance concern:** Any refactoring of `MarkdownElement` or `InlineSpan` enums (adding a field, renaming a variant) breaks all snapshots simultaneously. The team must resist the habit of auto-accepting (`cargo insta accept`) without review.

### 3.2 Performance Benchmarks (criterion)

**Priority:** P3
**Effort:** Medium (half day)
**Impact:** Prevents future performance regressions

Add `criterion` as a dev-dependency. Create 3-4 benchmark cases:
1. Parse a large markdown file (10K lines of mixed content)
2. Parse a table-heavy file (50 tables, 20 columns each)
3. Parse with many code blocks (100 fenced blocks, various languages)
4. D2 Sugiyama layout with 50-100 nodes

**Do NOT** add wall-clock assertion tests (`assert!(elapsed < X)`) — these are flaky across hardware. Use `criterion` with `cargo bench` and track trends manually.

No benchmarks exist today. Even having baselines for future comparison is valuable.

### 3.3 Fuzz Testing (Local Only)

**Priority:** P3
**Effort:** Medium (1-2 days for setup)
**Impact:** Finds panics, hangs, and memory issues that structured tests miss

**Platform concern:** `cargo-fuzz` (libFuzzer) does not officially support Windows. Options:
- Run in WSL (recommended for this project)
- Use `bolero` crate as engine abstraction (supports libFuzzer, AFL, and honggfuzz)
- Run in CI on Linux only

**Fuzz targets (in priority order):**
1. `fuzz_parse_markdown` — Core parsing pipeline, most complex logic
2. `fuzz_d2_parse` — Pure function, trivial to fuzz, no dependencies
3. `fuzz_highlight_segments` — Unicode byte/char mapping is fragile
4. `fuzz_escape_table_pipes` — 150-line state machine with manual indexing

**Corpus seeds:** Embedded sample files, one example of each element type, edge cases (empty, whitespace-only, only `#`).

**Do NOT add to CI.** Run locally when idle CPU is available. Typical workflow: start a fuzz run overnight, review findings in the morning.

### 3.4 Mutation Testing (D2 Crate Only)

**Priority:** P3
**Effort:** Low (install cargo-mutants, run scoped)
**Impact:** Diagnostic — exposes weak test assertions in algorithmic code

Run `cargo mutants -p mdmdview-d2` scoped to the D2 crate only (~10K lines). Estimated runtime: 30-60 minutes. This is feasible as a periodic developer-initiated check.

**Do NOT run on the main crate** (39K lines, 8-minute test suite = CPU-days).

Highest-signal targets within D2:
- `layout_sugiyama.rs` (5 tests for 470 lines — many mutations will survive)
- `compiler.rs` (49.8% coverage — large untested surface)
- `shapes.rs` (54.9% coverage)

## Tier 4 — Deprioritized

These were considered and explicitly set aside.

### 4.1 Mutation Testing on Main Crate

**Why not:** At 39K lines with an 8-minute test suite, a full mutation run would take CPU-days. The same insights come from code review. If attempted, scope to `markdown_renderer.rs` parsing functions only (~1-2 hour runtime).

### 4.2 Testing the Test Runner Further

**Why not:** The `full_test.rs` binary is at 48% coverage, which is appropriate. The testable pure functions (diff_images, html_escape) are already well-tested. The untested parts are orchestration (shelling out to cargo, capturing screenshots) that cannot be meaningfully unit-tested. If the test runner produces wrong results, you see wrong results. "Testing a test runner with tests" offers diminishing returns.

### 4.3 Full Snapshot Conversion

**Why not:** Converting all 476 markdown_renderer tests to snapshots would be fragile and noisy. Any enum refactoring breaks all snapshots simultaneously. `cargo insta accept` becomes muscle memory, defeating the purpose. Selective application to ~30-40 weak tests is sufficient.

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Visual regression flaky on CI (font differences) | Bundle test fonts via `--test-fonts`; widen CI thresholds |
| proptest adds test runtime | 256 cases adds ~2-5 seconds; negligible |
| insta snapshots become auto-accepted noise | Apply selectively to ~30 tests only; review discipline |
| cargo-fuzz doesn't support Windows | Run in WSL or use bolero crate |
| CI cost grows with team | Linux jobs are 1x billing; optimize Windows jobs if needed |
| D2 crate testing strategy diverges from main | Treat as separate concern; focus on geometric/visual correctness |

## Devil's Advocate Challenges Incorporated

The following challenges from the devil's advocate shaped this plan:

1. **"Coverage ≠ quality"** — Plan focuses on catching real bugs (visual, platform) over maximizing coverage numbers. Truism tests are acknowledged but not explicitly targeted for removal (low risk, not worth the churn).

2. **"Are we solving real problems?"** — Tier 1 items directly address categories of bugs the project has actually experienced. Tier 3-4 items are acknowledged as defensive/theoretical.

3. **"Maintenance burden"** — Each tool assessed for 6-month prognosis. Mutation testing and full insta conversion deprioritized specifically due to maintenance cost. Visual regression CI rated highest for sustainability (infrastructure already exists).

4. **"You're not testing what breaks"** — PR CI and cross-platform CI are the two most impactful additions, neither of which was on the original 8-item list.

5. **"The D2 crate is a separate concern"** — Acknowledged. D2 gets its own testing strategy focused on geometric invariants and visual comparison, not coverage metrics.

## Success Criteria

After implementation, the testing system should:

1. **Catch regressions before merge** — Every PR runs tests on Windows and Linux
2. **Catch visual regressions** — 26 test cases validate rendering on every PR
3. **Cover algorithmic edge cases** — D2 Sugiyama has adequate test depth
4. **Detect parsing panics on adversarial input** — proptest runs on every `cargo test`
5. **Provide baseline benchmarks** — Performance regressions are detectable

The testing system should NOT:
- Take more than 25 minutes per PR
- Require constant snapshot maintenance
- Run mutation testing or fuzzing in CI
- Chase coverage metrics for their own sake
