# Tech Debt & Code Quality Report — mdmdview

**Date:** 2026-03-15
**Codebase:** mdmdview v1.14.2 (~39K lines Rust, 18 source files)
**Review Team:** Dead Code Hunter, Duplication Detective, Tech Debt Analyst, Correctness Inspector, Devil's Advocate
**Compiler Baseline:** Zero warnings, zero clippy lints

---

## Executive Summary

The mdmdview codebase is in **good health**. No dead code was found, the compiler and clippy are clean, and no critical bugs were confirmed after adversarial review. The primary improvement opportunity is **deduplication of the pikchr/d2 renderer pair**, which share ~85-100% identical infrastructure code. Beyond that, a handful of medium-severity code quality items would improve readability and maintainability.

The Devil's Advocate pass **downgraded or eliminated** 8 of 15 challenged findings. Several severity ratings from the initial sweep were inflated due to incorrect measurements (function lengths, field counts) or unfamiliarity with egui's idiomatic patterns (deferred mutations, centralized app state).

## Section 1: Confirmed Findings (Action Recommended)

### 1.1 Pikchr/D2 Renderer Duplication — MEDIUM

**What:** `pikchr_renderer.rs` (2,677 lines) and `d2_renderer.rs` (2,137 lines) share near-identical infrastructure: struct layout, worker threads, job/result channels, LRU caching, debounce logic, SVG rasterization pipeline, error rendering, Drop implementations, and test helpers.

**Duplicated components (100% or near-identical):**

| Component | Pikchr Lines | D2 Lines | Similarity |
|-----------|-------------|----------|------------|
| `TextureEntry` struct | 120-127 | 116-123 | 100% |
| `DiagramDebounce` struct | 158-166 | 154-162 | 100% |
| `svg_cache_key()` | 135-137 | 100-102 | 100% |
| `bucket_width()` / `bucket_scale()` | 142-152 | 107-117 | 100% |
| `worker_loop()` | 247-265 | 243-261 | 100% |
| `Drop` impl | 810-820 | 777-787 | 100% |
| `::new()` constructor | 208-237 | 204-233 | ~95% |
| `process_job()` | 269-313 | 265-313 | ~98% |
| `render_error_block()` | 781-807 | 745-774 | ~96% |
| `rasterize_svg()` | 329-448 | 329-434 | ~85% |
| Test helpers | 832-877 | 799-844 | ~95% |

**Estimated extractable lines:** ~1,200-1,500 (shared module)

**Devil's Advocate challenge:** "The duplication is intentional — renderers are independently feature-gated and may need to diverge." **Rebuttal:** After 6+ months of parallel existence, the two renderers have NOT diverged in infrastructure. Only the source→SVG step differs. The shared infrastructure is stable and unlikely to need per-renderer customization. A shared `DiagramRasterizer` module would not couple the features.

**Risk of consolidation:** MEDIUM — requires careful trait/generic design to avoid over-abstraction. The source-to-SVG step genuinely differs and should remain per-renderer.

### 1.2 `render_table_tablebuilder` Function Length — MEDIUM

**What:** `C:\language\mdmdview\src\markdown_renderer.rs:4483-4928` — a 445-line function handling the full table layout pipeline: column statistics, policy computation, width resolution, natural width estimation, sizing, and egui table construction.

**Devil's Advocate challenge:** "Is it actually complex, or just a sequential pipeline?" **Verdict:** It IS sequential, which mitigates the complexity concern. However, it contains a large inner closure (`render_table`) and per-row rendering logic that would benefit from extraction. The function is not dangerous, but it's the hardest part of the codebase to modify.

**Suggested approach:** Extract 2-3 inner functions (column width computation, header rendering, body row rendering) without disrupting the linear flow.

### 1.3 History Code Clarity — LOW

**What:** `C:\language\mdmdview\src\app.rs:1271-1292` — `push_history()` has an asymmetric branch that only updates `history_index` in the `else` branch (when cap is NOT hit):

```rust
if self.history.len() > self.max_history {
    self.history.remove(0);          // index NOT updated
} else {
    self.history_index = self.history.len();  // index updated
}
```

**Devil's Advocate verified:** The code is **correct by arithmetic coincidence** — when the cap is hit, `remove(0)` reduces `len()` by 1, which exactly cancels the prior `push()`, so `history_index` (which was `old_len`) still equals `new_len`. But the structure is misleading.

**Suggested fix:** Unconditionally set `history_index = self.history.len()` after the cap check. Same behavior, clearer intent.

### 1.4 Mermaid JavaScript Patches — MEDIUM

**What:** `C:\language\mdmdview\src\mermaid_renderer.rs:1730-1760+` — hardcoded JavaScript replacement strings (300+ characters of minified code) that patch Mermaid.js internals. No comments explain what each patch does or how to verify it works.

**Risk:** If Mermaid.js is upgraded, patches may silently fail to match, producing broken rendering with no error. No automated verification that patches were applied.

**Suggested fix:** Add a comment above each patch explaining its purpose. Add an assertion or log warning if a patch target string is not found.

## Section 2: Findings Challenged and Downgraded

These findings were raised by the initial sweep but **debunked or significantly downgraded** by the Devil's Advocate after code verification.

### 2.1 "NaN panic in image cache sort" — ELIMINATED

**Original claim (HIGH):** `markdown_renderer.rs:613` — `partial_cmp().unwrap_or()` panics on NaN.
**Devil's Advocate:** `unwrap_or(Ordering::Equal)` returns the fallback on `None` — it does NOT panic. The code correctly handles NaN by treating it as equal. **Not a bug.**

### 2.2 "Off-by-one in backward search" — ELIMINATED

**Original claim (MEDIUM):** `app.rs:607-624` — backward search fails on single-element documents.
**Devil's Advocate:** Traced all code paths. Single-element case: `start = 0`, wraps to `total-1 = 0`, searches `(0..=0).rev()` = `[0]`, finds match correctly. Multi-element wrap-around also works. **Not a bug.**

### 2.3 "LRU Cache O(n) touch is CRITICAL" — Downgraded to LOW

**Original claim (CRITICAL):** `lru_cache.rs:103-106` — `retain()` is O(n) in hot path.
**Devil's Advocate:** Cache capacities are 64 entries. `retain()` over 64 elements costs microseconds. Even at 60fps with multiple diagrams, this is unmeasurable. `VecDeque::retain` on 64 items has better cache locality than a `LinkedHashMap` lookup. Not a performance problem at current scale. An `indexmap` would be more idiomatic but not faster at this size.

### 2.4 "`render_to_ui` is 362 lines" — ELIMINATED

**Original claim (CRITICAL):** Massive monolithic rendering function.
**Devil's Advocate:** The function is actually **45 lines** (`markdown_renderer.rs:2939-2984`). It's a clean loop calling `render_element_body` per element plus viewport distance tracking. The tech debt analyst likely measured the wrong function or included surrounding code. **Factually wrong.**

### 2.5 "MarkdownViewerApp has 94+ fields (God Object)" — Downgraded to LOW

**Original claim (HIGH):** 94+ fields, impossible to understand.
**Devil's Advocate:** Actual field count is **38** (`app.rs:370-464`). While large, this is standard for egui apps — immediate-mode GUIs require centralized state in the `App` struct. The fields are well-documented with comments. Some logical grouping (e.g., `SearchState`, `NavigationState`) could improve readability but wouldn't change the borrow-checker constraints.

### 2.6 "Deferred state mutations are tech debt" — ELIMINATED

**Original claim (HIGH):** Scattered deferred flags are a code smell.
**Devil's Advocate:** Deferred mutations are **idiomatic egui**. Inside `ctx.input_mut()`, you cannot call `&mut self` methods due to borrow conflicts. Setting flags and processing them after the input closure is the standard pattern documented in egui's own examples. **Not tech debt — correct usage.**

### 2.7 "Magic numbers (Flesch-Kincaid)" — ELIMINATED

**Original claim (MEDIUM):** Hard-coded values without explanation.
**Devil's Advocate:** The Flesch-Kincaid coefficients (`206.835`, `1.015`, `84.6`, `0.39`, `11.8`, `15.59`) are documented with formula comments directly above the lines. These are published constants from a well-known readability algorithm. **Not magic numbers.**

### 2.8 "MermaidTheme panics on Auto" — ELIMINATED

**Original claim (LOW):** Public methods panic on `Auto` variant.
**Devil's Advocate:** Every call site resolves `Auto` to a concrete theme before calling these methods. A `#[should_panic]` test documents this invariant. The panic is deliberate invariant enforcement, like `unreachable!()`. **Not a defect.**

## Section 3: Observations (No Action Needed)

### 3.1 Dead Code — Clean
The Dead Code Hunter found zero dead code. All functions have callers, all types are used, all feature flags are active, all enum variants are matched. The codebase is well-maintained.

### 3.2 Compiler Status — Clean
Zero compiler warnings. Zero clippy lints. Zero `#[allow]` attributes on production code (only on test report structs in `full_test.rs`).

### 3.3 Test Infrastructure
Heavy use of thread-local `RefCell` for test dependency injection (~20+ thread-locals). This is unconventional but works correctly and avoids adding runtime overhead for test-only abstractions. The Tech Debt Analyst flagged this as a smell, but the Devil's Advocate noted it's a deliberate design choice to avoid mocks.

### 3.4 `handle_shortcuts` Length (221 lines)
The shortcut handler is long due to repetitive patterns but highly readable and debuggable. Each shortcut is explicit and searchable. A data-driven approach might reduce line count but would make conditional shortcuts (raw-edit mode guards) harder to express. **Low priority.**
