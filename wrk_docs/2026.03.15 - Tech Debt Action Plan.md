## Phase 4: Future Considerations (Not Scheduled)

These items are noted for awareness but **not recommended for immediate action**. They represent design preferences rather than defects.

### 4.1 App state grouping
- Group `MarkdownViewerApp`'s 38 fields into logical sub-structs (`SearchState`, `NavigationState`, etc.)
- **Why deferred:** egui's `&mut self` constraint means sub-structs only help with readability, not borrow-splitting. Low ROI until the field count grows further.

### 4.2 LRU cache data structure
- Replace `VecDeque` + `HashMap` with `indexmap::IndexMap` for O(1) touch
- **Why deferred:** At n=64, the current implementation is fast enough. Only revisit if cache capacities are raised above ~500.

### 4.3 Data-driven shortcut table
- Replace 221-line `handle_shortcuts` with a table-driven approach
- **Why deferred:** Current approach is readable, searchable, and supports conditional shortcuts. Table-driven would reduce lines but add indirection. No defect risk from current approach.

### 4.4 `DiagramRenderer` trait unification
- Full trait-based polymorphism for pikchr/d2/mermaid renderers
- **Why deferred:** Phase 2 captures most of the value (shared infrastructure) without the abstraction overhead. A trait would only help if we need to treat renderers generically at runtime, which we currently don't.
